Elevação & Sombras

5 níveis de elevação Material 3 para criar hierarquia visual entre superfícies, botões, cards e barras de navegação.

Níveis de Elevação

elevation: 1
App/Button
Botões em estado padrão
elevation: 2
App/Card
Cards e containers
elevation: 3
App/FAB
Floating Action Button
elevation: 4
App/App Bar
Barra superior de navegação
elevation: 5
App/Button 2
Botões em destaque máximo

Uso correto

✅ Use
Eleve cards (2) acima do background de tela para criar separação visual clara entre superfícies.

Use elevation: 4 no AppBar para destacar o header fixo do conteúdo scrollável.
❌ Evite
Não use elevation: 5 em cards comuns — reserve para ações de alta prioridade e FABs.

Não empilhe sombras (um container dentro de outro com sombra), isso cria ruído visual.

Código Flutter

DART // Card com elevação padrão
Card(
  elevation: 2,
  shape: RoundedRectangleBorder(
    borderRadius: BorderRadius.circular(AppSpacing.borderRadiusCard),
  ),
  child: Padding(
    padding: EdgeInsets.all(AppSpacing.paddingCard),
    child: /* conteúdo */,
  ),
)

Do & Don't

✅ Use
Use apenas um botão Primary por tela — ele guia a atenção do usuário à ação mais importante. Combine com Secondary ou Ghost para ações de apoio.
❌ Evite
Não use dois botões Primary lado a lado. Isso dilui a hierarquia de ação e confunde o usuário sobre o próximo passo.

M3 define dois estilos de text field. O Konsi DS adota exclusivamente o Outlined por maior clareza visual em formulários financeiros.

M3Konsi DSObservação
Filled text field—Não utilizado — background preenchido prejudica leitura em formulários densos
Outlined text fieldInput/OutlinedPadrão Konsi. Borda 1dp → 2dp no foco, label flutuante M3

Acessibilidade

Label sempre visível. Não usar apenas placeholder — a label flutuante garante contexto mesmo após o preenchimento.
Error state. Sempre use ícone + texto de erro. Nunca comunique erro apenas via cor.

Quando Usar

Chips são elementos secundários. Prefira botões para ações primárias e checkboxes para seleção em formulários.

✅ Use chips quando…
  • Há 3 ou mais opções de filtro (≤2 opções → use checkboxes)
  • O usuário precisa adicionar/remover tags livremente (Chip/Input)
  • A ação é contextual e secundária — "Simular", "Ordenar por prazo"
  • O estado selecionado precisa ser visualmente persistente (Filter)
  • Precisa indicar status de um objeto de forma compacta (Status)
❌ Evite chips quando…
  • A ação é primária ou destrutiva — use Button
  • Serve como item de navegação — use Tab ou NavigationBar
  • Há apenas 1–2 opções — use Switch ou Checkbox
  • O texto do chip tem mais de 3–4 palavras
  • Tipos diferentes de chip estão misturados no mesmo grupo
💡
Chip vs Botão: Chips representam entidades (filtros, tags, estados). Botões disparam ações únicas e diretas. Se a label for um verbo de ação principal, use Button.

Como Contribuir

O Konsi DS é vivo e evolui com o time. Veja como propor melhorias, documentar novos componentes e corrigir inconsistências.

Processo de Contribuição

1
Identifique a lacuna
Encontrou um componente sem documentação, uma cor faltando ou uma inconsistência? Registre no backlog do time de Design com uma descrição clara do problema.
2
Discuta com o time
Antes de propor uma solução, alinhe com pelo menos um designer sênior ou tech lead. Mudanças no DS afetam todos os produtos da Konsi.
3
Documente a proposta
Crie uma proposta com: (a) problema identificado, (b) solução proposta com exemplos visuais, (c) impacto em componentes existentes.
4
Revisão e aprovação
A proposta passa por revisão do time de Design. Se aprovada, é adicionada ao Figma, ao código Flutter e a esta documentação.
5
Comunique o time
Após a publicação, comunique a mudança com changelog claro: o que mudou, por quê, e como migrar se necessário.

Princípios para Contribuição

Prefira evoluir o que existe a criar do zero. Novos tokens e componentes aumentam a carga cognitiva do time. Antes de propor algo novo, verifique se o problema pode ser resolvido com o que já está no DS.
